# gstack Fork Maintenance Guide

This document outlines the recommended workflow for maintaining your personal fork of gstack and syncing it with the upstream repository while preserving your custom modifications.

## Recommended Workflow: Feature Branches

Do not commit your custom modifications directly to the `main` branch. Instead, keep `main` strictly as a mirror of the upstream repository, and put your changes in feature branches.

### 1. Setting Up the Upstream Remote

```bash
git remote add upstream https://github.com/garrytan/gstack.git
git remote -v
```

### 2. Creating a Feature Branch

```bash
git checkout -b feat/my-custom-host

# Stage only the files you changed — never use git add . (may pick up binaries)
git add hosts/myhost.ts hosts/index.ts .gitignore test/host-config.test.ts

git commit -m "feat: add native support for myhost"
git push origin feat/my-custom-host
```

### 3. Handling Autogenerated Files (`SKILL.md`)

`SKILL.md` files are generated from `.tmpl` templates by `bun run gen:skill-docs`. They are tracked in git and **will show as modified** in your working tree after any generation run.

**Do not commit these changes** — they will cause merge conflicts when syncing with upstream. Discard before committing:

```bash
git restore $(git ls-files '**SKILL.md' 'SKILL.md')
```

To permanently hide them from `git status` without touching `.gitignore`:

```bash
echo "**/SKILL.md" >> .git/info/exclude
echo "SKILL.md" >> .git/info/exclude
```

## Syncing with Upstream Updates

### Step 1: Update your local `main` branch

```bash
git checkout main
git fetch upstream
git pull upstream main
git push origin main
```

### Step 2: Rebase your feature branch

```bash
git checkout feat/my-custom-host
git rebase main
```

If `hosts/index.ts` conflicts, resolve it — never resolve conflicts on `SKILL.md` files directly. After resolving:

```bash
bun run gen:skill-docs --host all   # regenerate from templates
git add hosts/                       # stage resolved sources only
git rebase --continue
```

### Step 3: Force push the rebased branch

```bash
git push --force-with-lease origin feat/my-custom-host
```

## Adding a New Host

For `symlink-generic` hosts (the standard case), `setup` is fully config-driven — no bash changes needed. Short version:

1. Create `hosts/myhost.ts` (copy from `hosts/kimi.ts`), set `linkingStrategy: 'symlink-generic'`
2. Import and add to `ALL_HOST_CONFIGS` in `hosts/index.ts`
3. Add `.myhost/` to `.gitignore`
4. Update host count assertion in `test/host-config.test.ts`
5. Run `bun run gen:skill-docs --host myhost && bun test`

`setup` detects and installs `symlink-generic` hosts automatically — no need to add gen sections, functions, or install blocks.

## Local Development & Linking

To use your fork's code instead of the globally installed version:

### Dev Mode (recommended)

Dev mode symlinks each skill's `SKILL.md` directly from your working tree so edits are live immediately — no rebuild or re-setup needed.

```bash
bin/dev-setup      # activate: your edits are live
bin/dev-teardown   # deactivate: back to global install
```

### Installing Skills for Specific Hosts

After editing `hosts/` or `setup`, regenerate and reinstall:

```bash
bun run gen:skill-docs --host myhost   # regenerate skill docs for one host
./setup --host myhost                  # install/re-link skills for that host
./setup --host auto                    # detect and install all found CLIs
```

Prefix preference is saved to `~/.gstack/config.yaml` on first run — no need to pass `--prefix` / `--no-prefix` again after the initial choice. To change it:

```bash
./setup --no-prefix   # switch to short names (/qa, /ship)
./setup --prefix      # switch to namespaced names (/gstack-qa, /gstack-ship)
```
